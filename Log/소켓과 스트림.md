소켓과 스트림은 TCP/IP 계층을 통해 데이터를 송수신한다.

# TCP/IP 4계층
1. 응용 계층: 개발자가 소켓을 통해 데이터를 쓰고 읽는 지점
2. 전송 계층: TCP(신뢰성) vs UDP(속도) 결정. 포트 번호를 통해 프로세스 식별
3. 인터넷 계층: IP 주소를 기반으로 최적의 경로를 찾는 라우팅 담당. 데이터를 패킷 단위로 포장
4. 네트워크 인터페이스: 물리적인 전기 신호(0, 1)로 변환하여 실제 전송

# 소켓(Socket)이란
소켓은 네트워크 통신의 양 끝점이다.
프로그램이 네트워크를 통해 데이터를 주고받기 위해 사용하는 통신창구이다.

소켓 = IP 주소 + 포트번호 + 통신방식(TCP/UDP)

소켓은 연결 → 통신 → 종료의 흐름을 담당한다.

소켓의 기본 역할
- 네트워크 연결 생성
- 데이터 송수신
- 연결 종료 관리

소켓 통신은 항상 클라이언트-서버 구조를 가진다.

소켓 통신의 기본 구조
서버
- 특정 포트에서 대기
- 클라이언트 연결 요청 수락
- 요청 처리 후 응답

클라이언트
- 서버의 IP와 포트로 연결 요청
- 데이터 전송 및 응답 수신

소켓의 분류
1. 네트워크 소켓
구분,TCP 소켓 (Stream 소켓), UDP 소켓 (Datagram 소켓)
특징,"연결 지향, 순서 보장, 재전송 있음","비연결형, 순서 없음, 빠른 속도"
비유,등기 우편/전화 (확인 절차 필수),일반 편지 (그냥 던짐)
용도,"웹(HTTP), 채팅, 파일 전송","실시간 스트리밍, 온라인 게임, DNS"

2. 유닉스 도메인 소켓 (Unix Domain Socket): 내 컴퓨터(같은 OS) 안에서 실행 중인 프로그램 간에 통신 시 사용
- 특징: 컴퓨터 안에서 통신할 때는 IP주소나 포트를 타고 네트워크 망을 거칠 필요가 없기 때문에 리눅스 파일 시스템(파일경로)를 이용해 다이렉트로 데이터를 주고받음
- 장점: 네트워크 헤더를 붙였다 떼는 과정이 생략되기 때문에 TCP 소켓보다 처리 속도가 훨씬 빠름
- 사용시기: Nginx 웹 서버와 백엔드 서버(WAS)가 같은 컴퓨터에 있을 때, 또는 도커(Docker) 데몬과 통신할 때 내부적으로 많이 쓰임

3. 웹소켓 (WebSocket): TCP소켓 활용
- 특징: HTTP 웹 통신으로 접속 후 실시간으로 연결 유지
- 사용시기: 웹 또는 앱에서 실시간 채팅, 푸시 알람 등의 기능 구현에 사용됨

# 스트림 (Stream): 가상의 파이프라인
표준 스트림 (Standart Stream): OS 데이터 통로
- 표준 입력 (stdin): 키보드로 들어오는 데이터 흐름 (예: System.in)
- 표준 출력 (stdout): 모니터 화면으로 나가는 데이터 흐름(예: System.out.println())
- 표준 에러 (stderr): 모니터 화면으로 나가는 에러 메시지 흐름(예: System.err.println())
 링크 - [Linux] 리눅스 기본 구조와 스트림

1. 특징: FIFO(First In First Out) 구조이며, 입력과 출력이 나뉜 단방향 통신
2. 분류
- 바이트 스트림: 1바이트 단위 (이미지, 동영상 등 바이너리) InputStream, OutputStream
- 문자 스트림: 2바이트 단위 (텍스트 파일) Reader, Writer
- 보조 스트림: 기본 스트림에 버퍼링(속도 향상) 등 기능을 추가. BufferedReader, PrintWriter

System.in(바이트 스트림)을 InputStreamReader(문자 스트림)로 감싸고, 다시 BufferedReader(보조 스트림)로 감싸는 구조

# 예제
1. 네트워크 소켓 (TCP/UDP)
// TCP 소켓 - 양방향 연결 후 데이터 전송
import java.io.*;
import java.net.*;

public class TcpExample {
    public static void main(String[] args) {
        // 서버 측 연결 대기 (포트 8080)
        try (ServerSocket serverSocket = new ServerSocket(8080)) {
            System.out.println("TCP 서버: 클라이언트 연결 대기 중...");
            Socket socket = serverSocket.accept(); // 연결 수락
            
            // 스트림을 이용한 양방향 통신
            PrintWriter out = new PrintWriter(socket.getOutputStream(), true);
            out.println("TCP 서버 접속을 환영합니다!"); // 데이터 전송 (순서와 도착 보장)
            
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}

// UDP 소켓 - 연결 없이 패킷만 주고받음
import java.net.*;

public class UdpExample {
    public static void main(String[] args) {
        // 전송 측
        try (DatagramSocket socket = new DatagramSocket()) {
            String message = "이것은 UDP 패킷입니다!";
            byte[] data = message.getBytes();
            InetAddress address = InetAddress.getByName("127.0.0.1"); // 목적지 IP
            
            // 데이터를 봉투(DatagramPacket)에 담아 던짐
            DatagramPacket packet = new DatagramPacket(data, data.length, address, 8080);
            socket.send(packet);
            System.out.println("UDP 데이터그램 발송 완료!");
            
        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}

2. 유닉스 도메인 소켓 (Unix Domain Socket) - 자바 버전 16부터 지원
import java.net.*;
import java.nio.channels.*;
import java.nio.file.Path;

public class UnixDomainSocketExample {
    public static void main(String[] args) {
        // IP나 포트 대신 리눅스 파일 시스템의 '경로'를 사용합니다.
        Path socketPath = Path.of("/tmp/my_app.sock");
        UnixDomainSocketAddress address = UnixDomainSocketAddress.of(socketPath);

        try (ServerSocketChannel serverChannel = ServerSocketChannel.open(StandardProtocolFamily.UNIX)) {
            serverChannel.bind(address);
            System.out.println("유닉스 도메인 소켓: 동일 OS 내 통신 대기 중...");
            
            // 이후 로직은 일반 소켓 채널과 동일하게 처리 (네트워크를 타지 않아 매우 빠름)
            
        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}

3. 웹소켓 (WebSocket) API
import javax.websocket.*;
import javax.websocket.server.ServerEndpoint;

// HTTP로 접속 후 ws:// 경로로 업그레이드하여 실시간 연결 유지
@ServerEndpoint("/chat")
public class WebSocketExample {

    @OnOpen
    public void onOpen(Session session) {
        System.out.println("웹소켓 연결 성공! 세션 ID: " + session.getId());
    }

    @OnMessage
    public String onMessage(String message, Session session) {
        System.out.println("클라이언트로부터 받은 메시지: " + message);
        return "서버 응답: 실시간 채팅 메시지 수신 완료!";
    }
}

4. 스트림 (Stream)
import java.io.*;

public class StreamWrapperExample {
    public static void main(String[] args) {
        // 1. 표준 입력 (OS가 제공하는 키보드 입력 데이터 통로, 바이트 스트림)
        InputStream standardIn = System.in;

        // 2. 바이트 스트림을 문자 스트림으로 변환 (1바이트 -> 2바이트로 묶어 한글 깨짐 방지)
        InputStreamReader charReader = new InputStreamReader(standardIn);

        // 3. 보조 스트림으로 감싸기 (버퍼를 달아서 속도를 높이고, readLine() 같은 편리한 기능 추가)
        BufferedReader bufferedReader = new BufferedReader(charReader);

        try {
            System.out.print("아무 문자열이나 입력해보세요: ");
            
            // 보조 스트림의 기능으로 데이터를 '한 줄(Line)' 단위로 편하게 읽음
            String inputData = bufferedReader.readLine();
            System.out.println("표준 출력(stdout)으로 내보내기: " + inputData);
            
        } catch (IOException e) {
            // 표준 에러(stderr)로 에러 메시지 출력
            System.err.println("입력 중 오류가 발생했습니다.");
        }
    }
}

과거 TCP 소켓에서는 바이트로 들어오는 데이터를 개발자가 직접 InputStream을 열고 보조 스트림을 달아 한 줄씩 읽어야 했지만 
자바 웹소켓 라이브러리나 프레임워크를 사용하면, 내부적으로 스트림 처리를 다 해주기 때문에 Stream이나 Reader를 직접 작성하지 않아도 활용이 가능하다.

# 소켓과 스트림 종료처리
통신이 끝난 소켓과 스트림은 close() 메서드로 반드시 자원을 반환해야 한다. 자원을 반환하지 않으면 OS 자원이 고갈되어 Too many open files 에러로 서버가 다운될 수 있다.

포트 좀비화 (Port 점유): 소켓이 완전히 닫히지 않아 포트를 계속 물고 있게 되며, 서버 재시작 시 Address already in use 에러가 발생한다.
고로 소켓과 스트림을 사용하는 구문에는 try-with-resources 문법을 사용하여 자원이 안전하게 반환되도록 처리하여야 한다.

웹소켓 프레임워크(Spring WebSocket 등)는 복잡한 원시 소켓을 Session(세션) 객체로 추상화하여 제공한다.
프레임워크가 알아서 닫는 경우: 사용자의 브라우저 종료나 네트워크 단절 시, 프레임워크가 알아서 소켓 자원을 해제하고 @OnClose 메서드를 실행한다.
개발자가 직접 닫아야 하는 경우: 강제 퇴장, 토큰 만료, 장기 미사용(Idle) 등 서버가 먼저 연결을 끊어야 할 때는 명시적으로 session.close()를 호출해야 한다.
@OnMessage
public void onMessage(String message, Session session) throws IOException {
    if (message.equals("bad_word")) {
        // 규정 위반 시 서버가 웹소켓 연결을 강제로 종료
        session.close(new CloseReason(CloseReason.CloseCodes.VIOLATED_POLICY, "욕설 금지"));
    }
}

세션 메모리 누수(OOM) 방지

문제 상황: 접속자 관리를 위해 만든 List<Session>이나 Map에 종료된 세션 객체를 그대로 방치하는 경우.
발생 결과: 죽은 세션 객체가 리스트에 여전히 참조되어 있어, 가비지 컬렉터(GC)가 메모리에서 지우지 못한다. 결국 접속자가 누적되면 서버 RAM이 꽉 차서 OutOfMemoryError(OOM)로 서버가 죽게 된다.
해결책: 사용자가 퇴장할 때 실행되는 @OnClose 메서드에서, 반드시 접속자 명단(List/Map)의 해당 세션을 remove()로 명시적 제거해야 한다.

프레임워크가 밑바닥의 지저분한 소켓 해제는 도와주지만, 논리적인 연결 객체(Session)의 생명 주기 관리는 오롯이 개발자의 몫이다. 물리적 연결이 끊어졌다고 해서, 내가 만든 리스트에서도 알아서 지워지는 것은 아니다

# 정리
1. 소켓은 네트워크의 연결문이고, 스트림은 문 사이로 흐르는 데이터의 물줄기
2. TCP는 꼼꼼한 관리자(스트림 방식), UDP는 빠른 배달원(데이터그램 방식)
3. 현대 개발에서는 고수준 프로토콜(HTTP 등)을 주로 쓰지만, 고성능 게임이나 IoT에서는 여전히 소켓 제어가 필수
4. 통신이 끝난 소켓과 스트림은 close() 메서드로 자원을 반환해야 함
