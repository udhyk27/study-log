# 네트워크 개념

🌐 네트워크 핵심 용어 한방 정리
1️⃣ IP 주소

👉 네트워크에서 기기를 식별하는 주소

내부 IP: 192.168.x.x (집/회사)
공인 IP: 인터넷에서 보이는 주소
2️⃣ 서브넷 마스크

👉 IP에서 네트워크 / 기기 구분 기준

예:

255.255.255.0
→ 앞 3칸 = 네트워크, 마지막 = 기기
3️⃣ 게이트웨이 (Gateway)

👉 다른 네트워크로 나가는 출입구

보통 공유기 주소 (192.168.1.1)
인터넷 나갈 때 무조건 거침
4️⃣ DNS

👉 도메인을 IP로 바꿔주는 시스템

예:

google.com → 142.x.x.x
5️⃣ 포트 (Port)

👉 하나의 IP 안에서 서비스 구분 번호

예:

80 → HTTP
443 → HTTPS
3306 → MySQL

👉 IP = 건물
👉 포트 = 방 번호

6️⃣ 프로토콜 (Protocol)

👉 통신 규칙

대표:

HTTP / HTTPS (웹)
TCP / UDP (전송 방식)
7️⃣ TCP vs UDP
TCP
연결형 (신뢰성 높음)
느림
예: 웹, 로그인
UDP
비연결형 (빠름)
신뢰성 낮음
예: 게임, 스트리밍
8️⃣ NAT

👉 내부 IP → 공인 IP로 변환

공유기가 수행
여러 기기가 1개의 공인 IP 사용 가능
9️⃣ DHCP

👉 IP 자동 할당 시스템

공유기가 자동으로 IP 줌
수동 설정 필요 없음
🔟 MAC 주소

👉 네트워크 카드의 고유 물리 주소

하드웨어에 박혀 있음
IP랑 다름
1️⃣1️⃣ 라우터 / 공유기

👉 네트워크를 연결해주는 장치

내부 ↔ 인터넷 연결
NAT, DHCP 기능 수행
1️⃣2️⃣ 방화벽 (Firewall)

👉 허용/차단 규칙

예:

특정 포트 막기
특정 IP 차단
💻 실무에서 진짜 자주 쓰는 조합
🔧 서버 접속 안 될 때

확인 순서:

IP 맞냐?
포트 열려있냐?
방화벽 막혀있냐?
DNS 제대로 되냐?
🌐 웹 서비스 구조
도메인 → DNS → 공인 IP → 공유기(NAT) → 내부 서버(IP:PORT)
🎯 핵심 5개만 기억해도 됨

👉 실무 필수 TOP 5:

IP
서브넷 마스크
게이트웨이
DNS
포트
🧠 초간단 비유 (암기용)
IP 👉 집 주소
서브넷 👉 동네
게이트웨이 👉 출입구

DNS 👉 전화번호부
🎯 최종 한 줄

👉 네트워크 = “IP(주소) + 포트(서비스) + DNS(이름) + 게이트웨이(출입)” 구조

# 네트워크 전체 흐름

👉 도메인 입력 → DNS 조회 → IP 획득 → 서버 연결 → 요청/응답 → 화면 표시

🧠 단계별 실제 통신 흐름
1️⃣ 사용자가 주소 입력

예:

https://google.com

👉 아직 IP 모름

2️⃣ DNS 조회

👉 DNS 서버에 물어봄

“google.com IP 뭐야?”

결과:

142.x.x.x
3️⃣ 서버 위치 결정

👉 이제 목적지 확정

IP: 142.x.x.x
포트: 443 (HTTPS)
4️⃣ TCP 연결 (3-way handshake)

👉 서버랑 연결부터 만듦

순서:

SYN (나 연결할래)
SYN-ACK (오케이)
ACK (연결 완료)

👉 이걸로 통신 채널 생성

5️⃣ HTTPS면 보안 연결 (TLS)

👉 여기서 암호화 시작

인증서 확인
키 교환 (RSA/ECC)
이후 데이터는 AES로 암호화
6️⃣ HTTP 요청 전송

브라우저 → 서버:

GET / HTTP/1.1
Host: google.com

👉 “메인 페이지 줘”

7️⃣ 서버 처리

서버 내부에서:

로직 실행
DB 조회
결과 생성
8️⃣ HTTP 응답

서버 → 브라우저:

200 OK
(html 데이터)
9️⃣ 브라우저 렌더링

👉 HTML, CSS, JS 해석해서 화면 그림

🔁 중간에 실제 네트워크 흐름 (중요)
내 PC → 공유기 → 인터넷 → 서버 → 다시 돌아옴

이때:

내부 IP → NAT → 공인 IP로 변환
여러 라우터 거쳐서 서버 도달
📦 핵심 요소들이 어디 쓰였냐
개념	어디서 등장
DNS	2단계
IP	3단계
포트	3단계
TCP	4단계
TLS	5단계
HTTP	6~8단계
NAT	이동 중
🎯 진짜 핵심 요약

👉 DNS로 IP 찾고 → TCP로 연결 만들고 → HTTPS로 암호화해서 → HTTP로 데이터 주고받는다

💡 실무에서 이게 왜 중요하냐

문제 생기면 이 순서대로 끊김:

DNS 문제 → 도메인 안 열림
TCP 문제 → 연결 실패
TLS 문제 → 인증서 에러
서버 문제 → 500 에러

👉 그래서 개발자는 이 흐름을 기준으로 디버깅함
