네트워크 개념

네트워크를 처음 공부하다 보면 용어가 많아서 헷갈리기 쉽습니다.
그래서 실무에서 자주 사용하는 개념 위주로 간단히 정리해보았습니다.

먼저 가장 기본이 되는 것은 IP 주소입니다. 네트워크에서 각 기기를 구분하기 위한 주소로, 집이나 회사에서 사용하는 192.168.x.x 형태의 내부 IP와 인터넷에서 보이는 공인 IP로 나뉩니다. 여기에 서브넷 마스크가 함께 사용되어 네트워크 범위와 개별 기기를 구분하게 됩니다. 예를 들어 255.255.255.0이라면 앞부분은 네트워크, 마지막은 기기를 의미한다고 이해하시면 됩니다.

외부 네트워크로 나갈 때는 게이트웨이를 거치게 됩니다. 일반적으로 공유기의 주소를 의미하며, 인터넷으로 나갈 때 반드시 통과하는 지점이라고 보시면 됩니다. 또한 우리가 사용하는 google.com과 같은 도메인은 바로 연결되는 것이 아니라 DNS를 통해 실제 IP로 변환된 뒤 통신이 이루어집니다.

포트 역시 중요한 개념입니다. 하나의 IP 안에서 여러 서비스가 동시에 동작하기 때문에 이를 구분하기 위한 번호입니다. 예를 들어 80은 HTTP, 443은 HTTPS와 같이 사용됩니다. 흔히 IP를 건물로 본다면 포트는 방 번호라고 이해하시면 편합니다.

통신 방식도 함께 알아두시면 좋습니다. HTTP나 HTTPS는 웹에서 사용하는 프로토콜이며, 실제 데이터 전송은 TCP나 UDP 방식으로 이루어집니다. TCP는 속도는 느리지만 안정성이 높고, UDP는 빠르지만 신뢰성이 낮은 특징이 있습니다. 그래서 웹 서비스나 로그인은 TCP를, 게임이나 스트리밍은 UDP를 사용하는 경우가 많습니다.

이 과정에서 공유기는 여러 역할을 수행합니다. 내부 IP를 공인 IP로 변환하는 NAT, 자동으로 IP를 할당하는 DHCP 기능 등을 함께 담당합니다. 또한 보안을 위해 방화벽이 특정 포트나 IP를 차단하기도 합니다. 참고로 MAC 주소는 네트워크 장비에 부여된 고유한 물리 주소로, IP와는 별개의 개념입니다.

실무에서는 모든 개념을 암기하기보다는 문제 상황에서 떠올릴 수 있는 것이 중요합니다. 예를 들어 서버 접속이 되지 않을 경우, IP가 올바른지, 포트가 열려 있는지, 방화벽이 차단하고 있는지, DNS가 정상인지 순서대로 확인하는 경우가 많습니다.

정리하자면 네트워크는 크게 “어디로(IP), 어떤 서비스로(포트), 어떻게 찾고(DNS), 어떻게 나가는지(게이트웨이)” 정도로 이해하시면 전체 흐름을 잡는 데 도움이 됩니다.

네트워크 전체 흐름

웹에서 주소를 입력했을 때 실제로 어떤 과정이 이루어지는지 이해하면 네트워크가 훨씬 쉽게 느껴집니다.

브라우저에 https://google.com을
 입력하면, 바로 서버에 연결되는 것이 아니라 먼저 DNS에 해당 도메인의 IP를 요청합니다. 이 과정을 통해 142.x.x.x와 같은 실제 IP를 전달받게 됩니다. 이 단계에서 문제가 발생하면 사이트 자체에 접속할 수 없습니다.

IP를 확인한 이후에는 해당 서버로 연결을 시도합니다. HTTPS의 경우 포트 443을 사용하게 됩니다. 이때 바로 데이터를 주고받는 것이 아니라, 먼저 TCP 방식으로 연결을 생성합니다. 흔히 3-way handshake라고 부르는 과정(SYN → SYN-ACK → ACK)을 통해 통신 채널을 확보하게 됩니다.

HTTPS 통신이라면 이후 TLS 과정을 통해 암호화 연결을 설정합니다. 인증서를 확인하고 안전한 통신을 위한 준비를 마친 뒤에야 실제 데이터 교환이 이루어집니다.

그 다음 브라우저는 서버로 HTTP 요청을 전송합니다. 서버는 이 요청을 받아 내부 로직을 실행하고, 필요에 따라 데이터베이스를 조회한 뒤 결과를 생성합니다. 이후 HTTP 응답으로 데이터를 반환하고, 브라우저는 이를 해석하여 화면을 구성하게 됩니다.

이 과정은 단순히 내 PC와 서버 간의 통신이 아니라, 실제로는 내 PC → 공유기 → 인터넷 → 서버를 거쳐 다시 돌아오는 구조입니다. 이때 NAT를 통해 내부 IP가 공인 IP로 변환되고, 여러 네트워크 장비를 거쳐 목적지까지 전달됩니다.

전체 흐름을 정리하면 다음과 같습니다.
DNS를 통해 IP를 확인하고 → TCP로 연결을 생성한 뒤 → HTTPS라면 암호화를 거치고 → HTTP로 데이터를 주고받습니다.

이 흐름을 이해하고 있으면 문제를 파악하는 데 큰 도움이 됩니다. 도메인이 열리지 않는 경우에는 DNS를, 연결이 되지 않는 경우에는 TCP를, 인증서 오류가 발생하면 TLS를, 서버 오류가 발생하면 애플리케이션 문제를 의심해볼 수 있습니다.

네트워크는 처음에는 복잡하게 느껴질 수 있지만, 이러한 흐름을 중심으로 이해하시면 전체 구조를 훨씬 쉽게 파악하실 수 있습니다.
