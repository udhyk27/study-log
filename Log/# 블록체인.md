## 1. 프로젝트 개요

음악 저작권 정산은 기존에 중앙화된 기관이 데이터를 독점 관리하는 구조로 운영되어왔습니다. 이 구조에서는 데이터 조작 가능성, 정산 지연, 투명성 부족 등의 문제가 발생할 수 있습니다.

이번 프로젝트는 **프라이빗 블록체인**을 활용해 음반 제작사, 협회, 음원사가 각자 노드를 운영하며 합의에 참여하는 구조를 설계했습니다. 어느 한 기관도 데이터를 단독으로 조작할 수 없고, 모든 정산 내역이 블록체인에 투명하게 기록됩니다.

### 핵심 기능

- 곡 등록 및 권리자별 지분율 설정
- 라이선스 구매 시 자동 정산
- 서버 사이드 지갑 생성 및 서명 (MetaMask 불필요)
- 승인 기반 수정 흐름 (추후 구현)
- 3개 기관 노드의 QBFT 합의를 통한 트랜잭션 검증

## 3. 시스템 아키텍처

```
사용자 (브라우저)
        │ HTTP
        ▼
Spring Boot 백엔드 (로컬 서버)
├── WalletService      지갑 생성 · ECDSA 서명
├── ContractService    ABI 인코딩 · calldata 생성
├── BesuService        JSON-RPC 통신 · receipt 폴링
└── MariaDB            트랜잭션 성공 후 DB 동기화
        │ JSON-RPC (eth_sendRawTransaction)
        ▼
VM1 Besu 노드 (10.84.255.101:8545)  ← 메인 RPC 연결점
        │ P2P (30303)
        ├── VM2 Besu 노드 (10.84.255.102)  협회
        └── VM3 Besu 노드 (10.84.255.103)  음원사
```

### 설계 원칙

**조회는 항상 DB에서**
트랜잭션 성공 후 이벤트를 파싱해서 MariaDB에 동기화합니다. 모든 조회는 DB에서만 처리하며, Besu RPC를 직접 조회하지 않습니다. 응답 속도와 안정성을 확보하기 위함입니다.

**서버 사이드 서명**
사용자 개인키를 프론트엔드에 노출하지 않습니다. 회원가입 시 서버에서 지갑을 자동 생성하고 AES-256으로 암호화해서 DB에 저장합니다. 트랜잭션 시 서버가 복호화해서 서명합니다.

### 6-2. WalletService — 지갑 생성 및 서명

회원가입 시 서버에서 Ethereum 규격 지갑을 자동 생성합니다.

```java
@Service
public class WalletService {

    // 지갑 생성
    public WalletResult createWallet() {
        // 1. 32바이트 랜덤 개인키 생성
        byte[] privateKeyBytes = new byte[32];
        new SecureRandom().nextBytes(privateKeyBytes);

        // 2. secp256k1 공개키 도출
        ECKeyPair keyPair = ECKeyPair.create(privateKeyBytes);

        // 3. keccak256으로 지갑 주소 도출
        String address = Keys.getAddress(keyPair);

        // 4. AES-256 암호화 후 DB 저장
        String encryptedKey = aesEncrypt(Numeric.toHexStringWithPrefix(keyPair.getPrivateKey()));

        return new WalletResult("0x" + address, encryptedKey);
    }

    // 트랜잭션 서명
    public String signTransaction(RawTransaction tx, String encryptedPrivateKey) {
        String privateKey = aesDecrypt(encryptedPrivateKey);
        ECKeyPair keyPair = ECKeyPair.create(Numeric.toBigInt(privateKey));
        return Numeric.toHexString(TransactionEncoder.signMessage(tx, 84100L, keyPair));
    }
}
```

### 6-3. ContractService — ABI 인코딩

스마트 컨트랙트 함수 호출을 위한 calldata를 생성합니다.

```java
@Service
public class ContractService {

    private final String CONTRACT_ADDRESS;
    private final String ABI;

    // registerSong calldata 생성
    public String encodeRegisterSong(String title) {
        Function function = new Function(
            "registerSong",
            List.of(new Utf8String(title)),
            List.of(new TypeReference<Uint256>() {})
        );
        return FunctionEncoder.encode(function);
    }

    // setShares calldata 생성
    public String encodeSetShares(BigInteger songId,
                                   List<String> wallets,
                                   List<BigInteger> roles,
                                   List<BigInteger> shares) {
        Function function = new Function(
            "setShares",
            List.of(
                new Uint256(songId),
                new DynamicArray<>(Address.class, wallets.stream().map(Address::new).collect(Collectors.toList())),
                new DynamicArray<>(Uint8.class, roles.stream().map(Uint8::new).collect(Collectors.toList())),
                new DynamicArray<>(Uint256.class, shares.stream().map(Uint256::new).collect(Collectors.toList()))
            ),
            Collections.emptyList()
        );
        return FunctionEncoder.encode(function);
    }
}
```

### 6-4. BesuService — JSON-RPC 통신

```java
@Service
public class BesuService {

    private final String RPC_URL;

    // nonce 조회
    public BigInteger getNonce(String address) {
        EthGetTransactionCount result = web3j
            .ethGetTransactionCount(address, DefaultBlockParameterName.LATEST)
            .send();
        return result.getTransactionCount();
    }

    // 트랜잭션 전송
    public String sendRawTransaction(String signedTx) {
        EthSendTransaction result = web3j.ethSendRawTransaction(signedTx).send();
        return result.getTransactionHash();
    }

    // receipt 폴링 (최대 60초)
    public TransactionReceipt waitForReceipt(String txHash) throws Exception {
        int maxRetry = 60;
        for (int i = 0; i < maxRetry; i++) {
            Optional<TransactionReceipt> receipt = web3j
                .ethGetTransactionReceipt(txHash)
                .send()
                .getTransactionReceipt();

            if (receipt.isPresent()) {
                if (receipt.get().getStatus().equals("0x1")) {
                    return receipt.get();
                }
                throw new RuntimeException("Transaction failed");
            }
            Thread.sleep(1000);
        }
        throw new TimeoutException("Receipt polling timeout");
    }

    // 읽기 전용 호출
    public String ethCall(String calldata) {
        EthCall result = web3j.ethCall(
            Transaction.createEthCallTransaction(null, CONTRACT_ADDRESS, calldata),
            DefaultBlockParameterName.LATEST
        ).send();
        return result.getValue();
    }
}
```

### 6-5. 트랜잭션 처리 흐름 (Controller → Service)

```java
@PostMapping("/songs/register")
public ResponseEntity<?> registerSong(@RequestBody SongRequest req,
                                       HttpSession session) {
    // 1. 세션에서 사용자 조회
    Long userId = (Long) session.getAttribute("userId");
    User user = userRepository.findById(userId).orElseThrow();

    // 2. DB에서 개인키 복호화
    String encryptedKey = user.getPrivateKey();

    // 3. nonce 조회
    BigInteger nonce = besuService.getNonce(user.getWalletAddress());

    // 4. calldata 생성
    String calldata = contractService.encodeRegisterSong(req.getTitle());

    // 5. RawTransaction 생성
    RawTransaction tx = RawTransaction.createTransaction(
        nonce, BigInteger.ZERO, BigInteger.valueOf(100000),
        CONTRACT_ADDRESS, BigInteger.ZERO, calldata
    );

    // 6. ECDSA 서명
    String signedTx = walletService.signTransaction(tx, encryptedKey);

    // 7. 블록체인 전송
    String txHash = besuService.sendRawTransaction(signedTx);

    // 8. receipt 폴링
    TransactionReceipt receipt = besuService.waitForReceipt(txHash);

    // 9. 이벤트 파싱 후 DB 저장
    songService.saveFromReceipt(receipt, user);

    return ResponseEntity.ok(Map.of("txHash", txHash));
}
```

```
ResponseEntity.ok(songDto)
```

### 7-4. 3노드 QBFT 합의 흐름

```
VM1 트랜잭션 수신
    ↓
VM1이 블록 제안 (Proposer)
    ↓
VM2, VM3에 Pre-Prepare 메시지 전파
    ↓
VM2, VM3 검증 후 Prepare 메시지 응답
    ↓
2/3 이상 (2개 이상) Prepare 수집
    ↓
Commit 메시지 전파
    ↓
블록 확정 및 체인에 추가
    ↓
VM1, VM2, VM3 동일 블록 보유
```

### 문제 1: QBFT가 PoW로 실행되는 문제

**증상**
```
BesuControllerBuilder | TTD difficulty is not present, creating initial sync phase for PoW
```

**원인**
genesis.json에 `londonBlock`과 `zeroBaseFee` 설정이 누락되어 있었습니다. Besu가 QBFT를 인식하려면 London 하드포크 이상이 활성화되어 있어야 합니다.

**해결**
```json
"londonBlock": 0,
"zeroBaseFee": true,
```

위 두 줄을 genesis.json config에 추가하고 체인 데이터를 초기화한 후 재시작했습니다.

### 문제 2: host-allowlist로 인한 RPC 접근 차단

**증상**
```json
{"message": "Host not authorized."}
```

**원인**
`--host-allowlist`에 `localhost`가 빠져있었습니다. curl로 `localhost`로 요청하면 `127.0.0.1`이 아닌 `localhost` 문자열로 Host 헤더가 전송됩니다.

**해결**
```bash
--host-allowlist="localhost,127.0.0.1,10.84.255.101,10.84.255.1"
```

`localhost`와 `127.0.0.1`을 모두 명시했습니다.

### 문제 3: 노드 간 P2P 연결 실패

**증상**

```json
{"result": []}  // admin_peers 결과가 빈 배열
```

### 문제 4: 체인 데이터 불일치 오류

**증상**
```
Supplied genesis block does not match chain data stored in /home/webmaster/besu-node/node/data
```

**원인**
genesis.json 수정 후 기존 체인 데이터를 완전히 삭제하지 않아 불일치가 발생했습니다.

**해결**
key 파일을 보존하면서 체인 데이터만 삭제합니다.

```bash
pkill -f besu
rm -rf ~/besu-node/node/data/database
rm -rf ~/besu-node/node/data/caches
rm -f ~/besu-node/node/data/DATABASE_METADATA.json
rm -f ~/besu-node/node/data/LOCK
# key 파일은 절대 삭제하지 않음
```

> key 파일을 삭제하면 노드 주소가 바뀌어 extraData를 다시 생성해야 합니다.

### 문제 5: extraData 노드 주소 불일치

**증상**
QBFT 검증자 확인 시 예상과 다른 주소 반환

**원인**
체인 초기화 과정에서 실수로 key 파일이 삭제되어 노드 주소가 변경되었습니다.

**해결**
변경된 노드 주소를 재확인하고 extraData를 다시 생성했습니다.

```bash
cat ~/besu-node/node/data/nodeAddress

# toEncode.json 수정 후 재생성
besu rlp encode --from=toEncode.json --type=QBFT_EXTRA_DATA

# genesis.json extraData 교체 후 체인 데이터 초기화 및 재시작
```

### 문제 7: 단일 노드에서 블록 생성 안 됨

**증상**
블록 번호가 `0x0`에서 증가하지 않음

**원인**

**해결**
3개 노드를 모두 실행하고 P2P로 연결한 후 블록 생성을 확인했습니다.

## 9. 마치며

### 구현하면서 배운 것들

**블록체인 트랜잭션은 비동기**
트랜잭션 전송 후 즉시 결과를 알 수 없습니다. receipt 폴링 방식으로 최대 60초를 기다려야 하며, 향후 Laravel Queue처럼 Spring의 비동기 처리(`@Async` + CompletableFuture)로 전환할 계획입니다.

**노드 키 관리가 핵심**
`node/data/key` 파일이 노드의 신원입니다. 이 파일이 삭제되면 노드 주소가 바뀌고 genesis.json의 extraData를 재생성해야 하므로 네트워크 전체를 초기화해야 합니다. 운영 환경에서는 반드시 암호화 백업이 필요합니다.

**조회와 트랜잭션의 분리**
모든 조회를 DB에서 처리하는 설계 덕분에 Besu 노드 장애 시에도 조회 기능은 정상 운영됩니다. 트랜잭션만 영향을 받는 구조가 실서비스에서 중요한 안정성을 제공합니다.

**QBFT의 의미**
단순히 기술적 구현을 넘어, 음반 제작사·협회·음원사 3개 기관이 각자 노드를 운영하며 합의에 참여한다는 것은 어느 한 기관도 데이터를 단독으로 조작할 수 없다는 신뢰 구조를 만드는 것입니다. 이것이 프라이빗 블록체인의 실질적인 가치입니다.

### 향후 계획

- Spring Batch / Async 적용으로 트랜잭션 비동기 처리
- 승인 기반 수정 흐름 구현 (다중 서명)
- nonce 동시성 문제 해결 (Redis 분산 락)
- Besu 노드 systemd 서비스 등록으로 자동 재시작
- 방화벽 규칙 정교화 (Laravel 서버 IP만 8545 허용)
