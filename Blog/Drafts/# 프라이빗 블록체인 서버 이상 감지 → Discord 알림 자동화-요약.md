# 프라이빗 블록체인 서버 이상 감지 → Discord 알림 자동화

이전에 Hyperledger Besu 기반 프라이빗 블록체인을 구축했던 내용을 정리했었습니다.

VM 위에서 Besu 노드를 여러 개 구성해 운영하는 형태였는데, 테스트를 계속하다 보니 생각보다 노드가 자주 종료되거나 RPC 응답이 멈추는 경우가 발생했습니다.

특히 VM 자체가 종료되거나 Besu 프로세스가 죽으면 블록 생성이 멈추는데, 직접 서버에 접속해서 확인하기 전까지는 상태를 알기 어려웠습니다.

그래서 이번에는 **블록체인 서버 상태를 외부에서 감시하고, 문제가 생기면 Discord로 알림을 받을 수 있는 모니터링 시스템**을 구성해보았습니다.

단순 ping 체크 수준이 아니라:

- VM 자체 다운
- Besu RPC 응답 상태
- Peer 연결 상태
- 블록 생성 정지
- 디스크 사용량

까지 함께 확인하도록 구성했습니다.

## 전체 구조

```text
[블록체인 VM]
      ↓ 감시
[모니터링 VM]
      ↓ Webhook
[Discord]
```

핵심은 **모니터링을 반드시 별도 VM에서 수행한 점**입니다.

처음에는 블록체인 VM 안에서 자기 자신을 체크하려고 했습니다.

하지만 VM 자체가 종료되면 모니터링 스크립트도 같이 죽어버립니다. 결국 가장 중요한 “서버 다운” 상황을 감지하지 못하게 됩니다.

그래서 별도의 VM에서 외부 관점으로 블록체인 서버를 바라보는 방식으로 변경했습니다.

## 감시 항목

다음 항목들을 기준으로 상태를 체크하도록 구성했습니다.

| 항목 | 확인 방식 | 기준 |
|---|---|---|
| VM 다운 여부 | `ping` | 응답 실패 |
| RPC 상태 | `eth_blockNumber` | 응답 없음 |
| Peer 연결 상태 | `net_peerCount` | 2개 미만 |
| 블록 생성 정지 | 블록 번호 비교 | 180초 이상 변화 없음 |
| 디스크 사용량 | node_exporter | 80% 이상 |

Besu 관련 상태는 RPC(8545)를 통해 확인하고, 시스템 메트릭은 `node_exporter(9100)`를 사용해 분리했습니다.

블록 생성 정지 기준은 QBFT 블록 생성 시간이 약 60초였기 때문에 **3번 이상 생성되지 않는 상황(180초)** 을 이상 상태로 판단했습니다.

## 사전 확인

모니터링을 만들기 전에 가장 먼저 한 것은 **실제로 접근 가능한지 확인**하는 작업이었습니다.

### 1. ping 테스트

먼저 VM 자체가 네트워크상 도달 가능한지 확인했습니다.

```bash
ping -c 3 10.84.255.101
```

정상 응답이 온다면 VM 자체는 살아있는 상태입니다.

### 2. RPC 포트 확인

Besu RPC 포트(8545)가 열려 있는지도 확인했습니다.

```bash
nc -zv 10.84.255.101 8545
```

포트가 닫혀 있으면 RPC 호출 자체가 불가능합니다.

### 3. 실제 RPC 호출

마지막으로 실제 JSON-RPC 요청이 정상 응답하는지 확인했습니다.

```bash
curl -X POST \
-H "Content-Type: application/json" \
--data '{"jsonrpc":"2.0","method":"eth_blockNumber","params":[],"id":1}' \
http://10.84.255.101:8545
```

응답이 정상적으로 오면 RPC 통신까지 문제없는 상태입니다.

이 과정에서 Besu가 외부 접근을 허용하도록:

```text
--rpc-http-host=0.0.0.0
--host-allowlist
```

설정도 함께 확인했습니다.

기본 설정 그대로면 외부 VM에서 접근이 막혀 있을 수 있습니다.

## Discord Webhook 연동

알림은 Discord Webhook을 사용했습니다.

Discord 채널에 URL 하나를 발급받고, 문제가 발생하면 해당 URL로 POST 요청을 보내는 구조입니다.

Webhook 생성 방법은 다음과 같습니다.

1. Discord 채널 우클릭
2. 채널 편집
3. 연동(Integrations)
4. Webhook 생성
5. URL 복사

생성 후 아래처럼 테스트 요청을 보내 정상적으로 메시지가 오는지 확인했습니다.

```bash
curl -H "Content-Type: application/json" \
-d '{"content":"테스트 메시지"}' \
"WEBHOOK_URL"
```

채널에 메시지가 도착하면 정상입니다.

Webhook URL은 사실상 비밀번호와 비슷한 역할을 하기 때문에 코드에 직접 넣지 않고 별도 설정 파일(`config.env`)로 분리했습니다.

```bash
DISCORD_WEBHOOK_URL=...
```

권한도 함께 제한했습니다.

```bash
chmod 600 config.env
```

## 방화벽 설정

처음에는 node_exporter 포트를 그냥 열어두려고 했습니다.

하지만 `/metrics`에는 시스템 정보가 그대로 노출되기 때문에 전체 공개는 위험했습니다.

그래서 **모니터링 VM IP만 허용하는 방식**으로 구성했습니다.

```bash
sudo firewall-cmd --permanent \
--add-rich-rule='rule family="ipv4"
source address="10.84.255.210"
port port="9100"
protocol="tcp"
accept'
```

여기서 일반적인:

```bash
--add-port=9100/tcp
```

방식을 사용하면 모든 IP에서 접근 가능합니다.

모니터링 서버만 접근 가능하도록 `rich-rule`을 사용해 화이트리스트 방식으로 구성했습니다.

## NAT 환경 IP 문제

방화벽 작업 중 예상치 못한 문제가 하나 있었습니다.

모니터링 VM에서 보는 자기 IP와, 블록체인 VM이 실제로 인식하는 출발지 IP가 달랐습니다.

모니터링 VM에서는:

```bash
ip route get 10.84.255.101
```

결과가:

```text
10.0.2.15
```

처럼 보였는데, 실제 서버에서는 접근이 계속 차단됐습니다.

결국 `tcpdump`로 서버 측에서 패킷을 직접 확인했습니다.

```bash
sudo tcpdump -i any -n port 8545
```

확인해보니 실제 출발지 IP는:

```text
10.84.255.210
```

이었습니다.

VirtualBox NAT 환경에서 주소 변환이 일어나고 있었던 것입니다.

이후 방화벽 허용 IP를 실제 IP 기준으로 수정해 해결했습니다.

## node_exporter 설치 중 SELinux 문제

Rocky Linux에서 `node_exporter`가 실행되지 않는 문제도 있었습니다.

증상은 다음과 같았습니다.

```text
status=203/EXEC
Permission denied
```

권한도 정상이고 실행 파일도 존재했는데 systemd가 계속 실행을 거부했습니다.

원인은 SELinux 라벨 문제였습니다.

`/tmp`에서 압축 해제 후 이동했더니 실행 파일 라벨이 그대로 유지된 상태였습니다.

확인:

```bash
ls -lZ /usr/local/bin/node_exporter
```

해결:

```bash
restorecon -v /usr/local/bin/node_exporter
```

라벨을 복구한 뒤 정상 실행되었습니다.

Rocky Linux 계열에서는 단순 권한 문제 외에도 **SELinux 라벨까지 확인해야 한다는 점**을 다시 느꼈습니다.

## 모니터링 방식

스크립트는 cron으로 **매분 실행**되도록 구성했습니다.

하지만 매분마다 알림이 계속 오는 방식은 오히려 보기 어려웠습니다.

그래서 **상태가 바뀌는 순간에만 알림**이 오도록 처리했습니다.

예를 들어:

```text
정상 → 이상
→ 장애 알림

이상 → 정상
→ 복구 알림

정상 유지
→ 알림 없음
```

이전 상태를 파일에 저장해두고 현재 상태와 비교하는 방식입니다.

덕분에 실제 장애가 발생했을 때만 Discord 알림이 오도록 만들 수 있었습니다.

## 장애 시나리오 테스트

구축 후 실제 장애 상황도 테스트했습니다.

### 1. Besu 프로세스 종료

먼저 Besu 프로세스를 강제로 종료했습니다.

```bash
kill <PID>
```

결과적으로:

- RPC 응답 실패
- Peer 조회 실패
- 블록 생성 중단

상태가 감지되었고 Discord 알림이 정상 도착했습니다.

Besu 재시작 후에는 자동으로 복구 알림도 확인했습니다.

---

### 2. VM 자체 종료

가장 중요했던 테스트입니다.

```bash
shutdown -h now
```

VM 자체를 종료했습니다.

별도 VM에서 감시하고 있었기 때문에:

```text
VM ping 실패
```

를 즉시 감지했고 서버 다운 알림이 정상 도착했습니다.

모니터링 서버를 분리한 이유가 바로 이 케이스 때문이었습니다.

---

### 3. Peer 노드 일부 종료

Validator 노드 하나를 종료해 Peer 수가 감소하는 상황도 테스트했습니다.

QBFT는 3노드 중 2개만 살아 있어도 블록 생성이 가능했기 때문에:

```text
Peer 부족
→ 블록 생성은 정상
```

상태가 구분되어 감지되었습니다.

단순 서버 다운만 보는 것이 아니라, **블록체인 네트워크 상태까지 확인할 수 있다는 점**이 괜찮았습니다.

---

### 4. 디스크 임계값 초과

실제 디스크를 채우는 대신 임계값을 낮춰 테스트했습니다.

```text
80% → 20%
```

현재 사용량이 기준을 넘어서도록 만든 뒤 디스크 경고 알림이 정상적으로 오는지 확인했습니다.

## 결과

현재는 블록체인 서버에 문제가 생기면 약 1분 이내로 Discord 알림을 받을 수 있는 상태입니다.

단순히 서버가 살아있는지만 확인하는 수준이 아니라:

```text
VM 상태
+ RPC 상태
+ Peer 상태
+ 블록 생성 여부
+ 디스크 상태
```

까지 함께 체크하도록 구성했습니다.

운영 중 문제가 생겼을 때 어떤 부분이 이상인지 바로 확인할 수 있어 디버깅 시간도 많이 줄어들었습니다.
