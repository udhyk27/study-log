
## 만들게 된 이유

업무에서 탭 구분 텍스트 형태의 로그 파일을 주기적으로 정제해야 하는 일이 있었습니다. 파일마다 필드 수가 맞지 않는 행이 섞여 있었는데, 기존 도구들은 오류 행을 찾아주는 수준이었고 실제 수정은 대부분 수작업으로 진행해야 했습니다.

실제로 작업하다 보면 같은 수정이 수십 번, 많게는 수백 번 반복되는 경우가 있었습니다. 자동으로 판단할 수 있는 수준의 오류는 프로그램이 처리하고, 애매한 경우만 사용자가 확인하는 형태로 만들고 싶었습니다.

## 파일 구조를 어떻게 파악할 것인가

처음에 부딪힌 문제는 정상 필드 수를 어떻게 판단할 것인가였습니다.

파일 유형별로 config에 필드 수를 명시할 수 있도록 설계했지만, 그 전에 파일 자체만으로도 어느 정도 구조를 파악할 수 있지 않을까 생각했습니다.

그래서 전체 행의 필드 수 분포를 분석해 가장 많이 등장하는 값을 정상 기준으로 추론하도록 했습니다.

파일마다 컬럼 구조가 조금씩 달랐고, 같은 유형이라도 업체별로 구조가 다르게 들어오는 경우도 있었습니다. 그래서 config에 정의된 값이 있으면 우선 사용하고, 없으면 파일 자체에서 추론하는 형태로 동작하도록 구성했습니다.

## 오류 탐지

오류 유형은 크게 네 가지로 분류했습니다.

초과 필드와 누락 필드는 비교적 단순합니다. 기준 필드 수보다 많으면 초과 필드, 적으면 누락 필드로 판단합니다.

행 분리는 조금 다른 방식으로 처리했습니다. 개행 문자 때문에 하나의 행이 두 줄로 나뉘는 경우가 있었는데, 연속된 두 행을 합쳤을 때 정상 필드 수가 되면 분리된 행으로 판단했습니다.

가장 판단하기 어려웠던 건 형식 이상이었습니다.

형식 이상은 필드 수는 정상인데 값의 형태가 다른 경우입니다. 이를 위해 정상 행들을 기준으로 컬럼 프로파일을 생성했습니다. 주요 데이터 타입, 빈값 비율, 길이 분포 등을 분석한 뒤 해당 컬럼의 일반적인 특성을 저장해두고, 오류 행이 이 특성과 얼마나 다른지를 비교하는 방식입니다.

단순히 정규식 규칙만 사용하는 것보다 실제 데이터 분포를 기준으로 판단하는 편이 오탐이 적었습니다.

다만 이 부분은 비교적 보수적으로 설계했습니다. 2개 이상 컬럼에서 이상이 감지되면 오류로 분류하고, 1개 컬럼만 이상한 경우에는 의심 상태로만 표시하도록 했습니다.

## 자동수정 후보 생성

오류를 찾는 것 자체보다 어떻게 수정할지를 판단하는 쪽이 더 어려웠습니다.

예를 들어 필드가 하나 초과된 행이라면 어느 위치의 필드를 제거해야 정상 구조가 되는지 여러 경우를 계산해야 합니다.

그래서 오류 행마다 수정 후보를 여러 개 생성하고 각각 점수를 계산하도록 만들었습니다.

점수 계산에는 컬럼 프로파일을 사용합니다. 수정 결과가 정상 데이터 분포와 얼마나 유사한지를 평가하고 상위 3개 후보를 사용자에게 제시합니다.

사용자는 후보를 비교한 뒤 직접 선택할 수 있습니다.

또한 점수가 75점 이상인 후보만 일괄 자동수정 대상으로 처리하도록 했습니다.

이 기준은 실제 파일을 여러 번 테스트하면서 조정했습니다. 점수가 너무 낮으면 잘못된 수정이 섞이고, 너무 높으면 자동화 효과가 줄어들었습니다.

## 수동 편집과 Undo/Redo

자동수정만으로 모든 경우를 처리할 수는 없기 때문에 셀 직접 편집 기능도 함께 추가했습니다.

다만 수동 편집을 지원하면 반드시 되돌리기 기능이 필요했습니다. 수정하다가 실수한 경우 원래 상태로 복구할 수 있어야 했기 때문입니다.

그래서 QUndoStack을 사용해 최대 200단계까지 Undo/Redo를 지원하도록 했습니다.

셀 편집뿐 아니라 자동수정 적용, 일괄 자동수정, 필드 삽입 및 삭제, 행 삭제까지 모두 Undo/Redo 대상에 포함했습니다.

수정 검증 탭을 별도로 만든 이유도 비슷합니다.

수정이 많아질수록 어떤 행이 어떻게 바뀌었는지 확인하기 어려워졌습니다. 그래서 수정된 행들을 그룹 단위로 묶고 원본과 현재 값을 비교할 수 있도록 했습니다.

필요한 경우 특정 시점 상태로 복원하는 기능도 함께 넣었습니다.

## 학습 기반 추천

같은 파일 유형을 반복해서 처리하다 보면 비슷한 수정 패턴이 계속 나타났습니다.

예를 들어 특정 유형의 파일에서는 항상 같은 위치의 필드를 삭제하거나 특정 위치에 빈 필드를 추가하는 경우가 있었습니다.

그래서 사용자가 자동수정 후보를 승인하거나 수동으로 필드를 삽입 또는 삭제했을 때 해당 정보를 학습 데이터로 저장하도록 했습니다.

학습 데이터는 correction_patterns.json에 저장됩니다.

이후 같은 유형의 파일을 분석할 때 해당 위치의 후보 점수에 추가 보너스를 부여해 추천 순위에 반영하도록 했습니다.

문제는 학습 범위를 어디까지 같은 유형으로 볼 것인가였습니다.

처음에는 파일 유형과 필드 수만 기준으로 사용했습니다. 하지만 같은 유형이라도 업체마다 컬럼 구조가 다른 경우가 있었고, 학습 데이터가 서로 섞이면서 오히려 잘못된 추천이 발생하기 시작했습니다.

그래서 컬럼 프로파일을 기반으로 타입 시퀀스를 생성하고 이를 구조 시그니처로 사용했습니다.

예를 들어 `log|n=17|sig=ISSDSSSSSS` 형태의 스코프 키를 만들고, 구조가 다르면 자동으로 다른 학습 영역으로 분리되도록 했습니다.

## v1.0.5에서 학습 알고리즘을 수정한 이야기

학습 기능을 넣고 테스트를 진행하던 중 이상한 점을 발견했습니다.

학습 데이터는 계속 쌓이고 있었는데 실제 추천 결과에는 거의 변화가 없었습니다.

원인을 확인해보니 저장할 때 사용하는 키와 조회할 때 사용하는 키가 서로 달랐습니다.

저장은 컬럼 이름과 인접 타입을 조합한 컨텍스트 키로 하고 있었지만 조회는 정수 인덱스로 수행하고 있었습니다.

결국 저장은 되고 있었지만 실제 조회는 전혀 되지 않는 상태였습니다.

추가로 확인해보니 자동수정 후보를 승인했을 때 학습을 기록하는 함수 자체가 호출되지 않고 있었습니다.

관련 함수는 존재했지만 Undo 커맨드 쪽에서 연결이 빠져 있었습니다. 결과적으로 수동 수정만 기록되고 자동수정 승인 데이터는 전혀 학습되지 않고 있었습니다.

v1.0.5에서는 이 부분을 정리하면서 저장과 조회 방식을 모두 정수 인덱스 기반으로 통일했습니다.

구조 시그니처가 동일한 범위에서는 컬럼 위치 의미가 유지되기 때문에 인덱스만으로도 충분하다고 판단했습니다.

여기에 학습 초기 데이터 문제도 있었습니다.

기존에는 기록이 1건만 있어도 즉시 최대 보너스가 적용되는 구조였습니다. 우연히 발생한 수정 한 번이 이후 추천 결과에 영향을 줄 수 있는 상태였습니다.

그래서 최소 5건 이상 누적된 경우에만 학습 보너스를 적용하도록 변경했습니다.

학습 데이터 관리 화면도 함께 정리했습니다.

기존에는 내부 스코프 키와 컬럼 정보가 그대로 표시됐는데, 현재는 파일 유형과 "17열 제거 1,410회" 같은 형태로 보여주도록 수정했습니다.

## 대용량 파일 처리

실제 업무 파일 중에는 1GB가 넘는 경우도 있었습니다.

초기에는 파일 전체를 문자열로 읽어 처리했지만 성능이 좋지 않았습니다.

그래서 파일 읽기 방식을 4MB 바이너리 청크 단위로 변경하고 바이트 레벨에서 직접 파싱하도록 수정했습니다.

시스템 호출 횟수가 줄어들면서 처리 속도도 개선됐습니다.

UI 쪽도 별도 대응이 필요했습니다.

수백만 행을 모두 테이블에 표시하면 렌더링 자체가 어려웠기 때문에 화면에는 최대 20,000행까지만 표시하도록 했습니다.

대신 분석과 저장은 전체 데이터 기준으로 수행됩니다.

필터 기능에서도 문제가 있었습니다.

처음에는 화면에 표시된 20,000행만 기준으로 필터링하고 있었는데, 그 결과 나머지 데이터에 존재하는 오류 행은 확인할 수 없었습니다.

그래서 필터 전환 시 전체 행을 기준으로 다시 계산하도록 수정했습니다.

이 과정에서 화면 인덱스와 실제 데이터 인덱스를 분리하기 위해 display_map 구조를 도입했습니다.

## SFTP 기능을 넣은 이유

로그 파일이 서버에 있는 경우가 많아서 매번 수동으로 다운로드하는 과정이 번거로웠습니다.

그래서 프로그램 안에서 바로 서버에 접속해 파일을 내려받고 분석할 수 있도록 SFTP 기능도 추가했습니다.

로컬 파일이든 원격 파일이든 이후 분석 흐름은 동일하게 처리됩니다.

## 현재 구조

```text
app.py              진입점
src/core/           오류 탐지, 자동수정 후보, 학습기
src/ui/             UI 컴포넌트
src/utils/          파일 I/O, config 관리
config/config.json  파일 유형 정의
correction_patterns.json  학습 데이터
```

PyQt6 기반 데스크톱 애플리케이션이며 별도 데이터베이스 없이 JSON 파일로 학습 데이터를 관리하고 있습니다.

## 앞으로

현재는 학습 데이터가 실제 추천 품질 향상에 어느 정도 도움이 되는지 계속 확인하고 있습니다.

구조 시그니처 기반 분리 방식이 의도한 대로 동작하는지도 실제 데이터로 검증해볼 예정입니다.

기능적으로는 수정 이력을 history.jsonl 형태로 내보내는 기능과 컬럼별 타입 규칙을 직접 정의할 수 있는 편집기를 추가할 계획입니다.
