이더리움:

Fabric:

  Go SDK   → 가장 성숙 
  → Fabric 자체가 Go로 만들어져서
  → Go가 가장 안정적

솔라나:
  Rust → 가장 성숙 
  JS   → 성숙 
  Java → 약함️
  → 솔라나는 Java 쓰면 불편함

# 노드 접근 허용 방식 = enode 주소

1. 회사가 협력사에게 enode 주소 전달
   "우리 노드 주소는 이거야"
   enode://abc123...@192.168.1.1:30303

2. 협력사가 Besu 설치 후 접속
   besu --bootnodes="enode://abc123...@192.168.1.1:30303"

3. 허가 확인 (프라이빗이라 추가 설정)
   → 허가된 enode 목록에 있으면 접속 허용
   → 없으면 차단

4. 동기화 시작
   → 기존 블록 전부 다운로드
   → 이후 새 블록 실시간 공유

# 서비스 구조
관리자 (회사):
  → 네트워크 개설
  → 누가 참여할지 결정
  → 누가 validator할지 결정
  → 서버 열고 닫기
  → 어떤 기능 제공할지
  → 규칙 설정

validator 노드 (회사+협회 등):
  → 블록 생성/서명
  → 위변조 검증
  → 2/3 합의 참여
  → 신뢰할 수 있는 기관만

일반 참여 노드 (음원사 등):
  → 블록 데이터 저장
  → 데이터 조회
  → 트랜잭션 발생
  → 서명 권한 없음

사용자 (구매자/판매자):
  → MetaMask만 설치
  → 노드 아님
  → 서비스 이용만

# 새 블록 제안시
각 노드 = 각자 개인키 보유
        ↓
새 블록 제안 (협회가 데이터 변경 요청)
        ↓
회사 Besu:
  "이전 블록 해시값 확인 → 올바름 → 개인키로 서명"
음원사 Besu:
  "이전 블록 해시값 확인 → 올바름 → 개인키로 서명"
        ↓
2/3 서명 완료 → 블록 확정!

** 올바른 변경:
  협회가 정당한 데이터 변경 요청
  → 해시값 체인 올바름
  → 회사, 음원사 서명 
  → 블록 확정

** 위변조 시도:
  협회가 과거 블록 몰래 조작
  → 해시값 체인 불일치
  → 회사, 음원사 서명 거부 
  → 블록 확정 실패

# 블록체인 데이터는 절대 삭제 안됨!
컨트랙트 배포하면 고유의 주소 생성됨

0xAAAA... (주소)에 저장된 데이터
→ 재배포해도 삭제 X
→ 블록체인에 영원히 존재
→ 주소만 알면 언제든 조회 가능

단 우리가 DB 초기화(node1/data 삭제)하면
→ 블록 자체가 사라지므로
→ 0xAAAA... 도 같이 사라짐

# 허가 방법
방법1. 정적 노드 설정 (Static Nodes):
  static-nodes.json 파일에 허가된 enode 목록 작성

방법2. 온체인 허가 (Permissioning):
  블록체인 자체에 허가 목록 저장

방법3. VPN + 프라이빗 네트워크:
  VPN으로 같은 네트워크 묶고
  그 안에서만 Besu 운영

# 위변조 가능성
회사(노드) - 음원사(노드) - 협회(노드)

만약 협회가 자신들 노드에 있는 블록(데이터)를 위변조 시도하면
블록체인 Besu가 검증

-> 새 블록 받으면 Besu가 자동으로 검증:

1단계. 해시 검증
   이전블록 해시값이 맞는가?
   
2단계. 서명 검증
   허가된 검증자가 서명했는가?
   
3단계. 데이터 검증
   트랜잭션 형식이 올바른가?
   잔액이 충분한가?
   
4단계. 합의 검증 (IBFT2)
   2/3 이상 검증자 서명이 있는가?
        ↓
전부 통과 → 블록 수락 
하나라도 실패 → 블록 거부

# 합의 알고리즘
PoW (비트코인):
  → 가장 긴 체인을 정답으로 인정
  → 조작하려면 전체 51% 이상 컴퓨팅 파워 필요
  → 현실적으로 불가능 (천문학적 전기세)

PoS (이더리움 현재):
  → ETH를 많이 예치한 검증자들이 투표
  → 조작하려면 전체 ETH의 51% 보유해야 함
  → 현실적으로 불가능 (천문학적 돈)

IBFT2 (우리 Besu):
  → 검증자들이 2/3 이상 동의해야 블록 확정
  → 프라이빗이라 검증자가 우리 1명뿐
  → 미리 정해진 검증자들이 투표로 블록 확정
  → 프라이빗 네트워크에 적합
  → 빠르고 확정적

DPoS (Delegated PoS) - 위임지분증명:
  → EOS
  → 투표로 대표자 선출, 대표자가 블록 생성
  → 민주주의 방식
