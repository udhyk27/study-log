# Fabric vs Ethereum

1. 프라이버시 — 가장 중요한 이유
[Ethereum 프라이빗]
계약서 데이터 → 모든 노드에 동일하게 저장
→ 네트워크 참여자라면 누구나 조회 가능

[Hyperledger Fabric]
계약서 데이터 → Private Data Collection
→ 계약 당사자 (판매자 + 구매자) 노드에만 저장
→ 나머지 참여자는 "거래가 있었다"는 사실만 앎

2. 신원 인증 체계
[Ethereum]
지갑주소: 0x742d35Cc6634C0532...
→ 누구인지 알 수 없음
→ 법적 계약 당사자 특정 불가

[Fabric - MSP]
인증서 기반 신원: CN=김철수, OU=아티스트, O=음악저작권사
→ 실명 + 역할이 블록체인에 기록
→ 법적 효력 있는 전자서명 가능

3. Gas Fee 없음
[Ethereum]
계약 생성         → Gas 비용 발생
로열티 정산       → Gas 비용 발생
메타정보 업데이트 → Gas 비용 발생
→ 거래량 많을수록 운영비 폭증

[Fabric]
모든 트랜잭션 → 수수료 없음
→ 로열티 정산이 수천 건이어도 비용 없음

# 스마트 컨트랙트 언어
Ethereum  → Solidity
Fabric    → Go / JavaScript / Java (Chaincode)

                  Fabric              Ethereum(프라이빗)
──────────────────────────────────────────────────────
설정 난이도        높음 ★★★★★          중간 ★★★☆☆
PHP 연동 난이도    중간 (Gateway)      쉬움 (web3.php)
프라이버시         완벽                부족
신원인증           실명 기반           익명
거래 비용          없음                없음(프라이빗)
법적 적합성        높음                낮음
저작권 비즈니스    ✅ 최적             ⚠️ 부적합

# Fabric 방식 (다른 점)
트랜잭션이 있을 때만 블록 생성
→ 아무 거래 없으면 블록 안 만들어짐
→ 용량 증가가 훨씬 느림
→ 합의 알고리즘: Raft (Orderer 기반)
